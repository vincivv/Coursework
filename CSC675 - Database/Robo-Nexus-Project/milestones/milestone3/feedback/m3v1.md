# Feedback on Business Rules

Many of the rules you defined are valid at the system level, but they are not all database requirements/rules. Some of them are application oriented rules.

You are currently mixing three different layers:
1. Database constraints (what the database can enforce)
2. Application logic (what the backend or services should handle)
3. System/AI behavior (what external systems or algorithms decide)

For this course, I expect you to focus on database-level rules that can be grouped into business requirements.

Examples:

This is a database requirement:
- A robot can only have one active task at a time  
(can be enforced with constraints, triggers, or schema design)

This is not a database requirement:
- AI should recommend valid roles to the robot  
(this is application or AI logic)

This is also not a database responsibility:
- Collision avoidance between robots  
(this involves real-time spatial reasoning and should be handled outside the database)

What you need to do:
- Review each rule and decide whether it belongs in the database layer
- Keep rules that can be enforced using constraints, triggers, procedures, or schema design
- Move or remove rules that belong to application logic or AI behavior

The goal is to clearly define what the database is responsible for enforcing.
